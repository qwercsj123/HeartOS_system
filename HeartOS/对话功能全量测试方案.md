# HeartOS 对话功能全量测试方案

本文档面向 HeartOS 的整体验收与回归测试，不局限于某一个修复点，而是覆盖对话功能的完整能力面。

目标：

1. 验证对话是否能正确识别用户意图
2. 验证是否会把“问一问”误判成“直接执行”
3. 验证不同附件类型下的推荐、引导、执行是否正确
4. 验证多轮追问、结果解释、乱问、误问、边界输入时系统是否稳定
5. 验证 HeartOS 对平台边界的表达是否一致，不误导用户

---

## 1. 测试范围

当前对话功能主要覆盖以下能力：

- 平台身份问答
- 平台能力问答
- 使用方式引导
- 文件类型支持说明
- 手动数字化 / 自动数字化区别说明
- ECG 心电组学 / ECG 信号补全说明
- 图片/PDF 路由到数字化或风险评估
- XML/CSV/波形数据路由到心电组学或补全
- 已有结果的解释与追问
- 非平台相关闲聊、乱问、模糊问句的兜底响应

当前核心意图包括：

- `knowledge_qa`
- `chat`
- `ecg_auto_digitize`
- `ecg_manual_digitize`
- `ecg_feature_extract`
- `ecg_reconstruct`
- `zhunxin_risk_assess`
- `result_interpretation`

## 2. 测试原则

### 2.1 高优先级原则

优先保证以下事情绝不能出错：

- 询问句不能误触发执行
- 错文件类型不能被错误执行
- 结果追问不能误触发重新分析
- 用户乱问时系统不能崩、不能胡乱调用工具
- 平台不能表现成“替代医生诊断”

### 2.2 用户视角原则

测试时不要只站在研发视角，还要模拟真实用户：

- 知道自己要什么的用户
- 不知道平台怎么用的用户
- 只有图片、不知道下一步做什么的用户
- 会乱问、错问、重复问的用户
- 会带附件但说话不清楚的用户

### 2.3 多轮原则

每个能力至少要测：

- 单轮输入
- 连续追问
- 换说法重复问
- 带附件与不带附件

---

## 3. 测试前准备

建议准备以下来源文件：

### 3.1 图片类

- 标准 12 导联心电图图片 1 张
- 质量较差 / 模糊图片 1 张
- 非心电普通图片 1 张

### 3.2 PDF 类

- 心电图 PDF 1 份
- 非心电 PDF 1 份

### 3.3 波形类

- 标准 ECG CSV 1 份
- 标准 ECG XML 1 份
- TSV/TXT/JSON 波形文件各 1 份
- 非 ECG 普通 CSV 1 份

### 3.4 状态准备

- 无来源文件状态
- 已上传但未勾选状态
- 单个来源已勾选状态
- 多个来源同时勾选状态
- 已产生工具结果后的状态

---

## 4. 测试维度

建议从这 8 个维度做矩阵测试：

1. 用户意图类型
2. 来源文件类型
3. 是否已勾选来源
4. 是否明确表达“要执行”
5. 是否只是问能力/问能否
6. 是否是结果追问
7. 是否多轮上下文
8. 是否异常/乱问输入

---

## 5. 功能测试场景

## 5.1 平台身份与能力问答

### 场景 5.1.1：身份问答

输入示例：

- `你是谁`
- `你是做什么的`
- `介绍一下你自己`

期望结果：

- 回答平台身份
- 不提及底层模型或外部品牌
- 不触发任何工具

### 场景 5.1.2：能力问答

输入示例：

- `你能做什么`
- `你支持什么功能`
- `HeartOS 能处理哪些心电任务`

期望结果：

- 回答平台支持的核心能力
- 不触发任何工具

### 场景 5.1.3：平台介绍

输入示例：

- `HeartOS 是什么`
- `这个平台是干什么的`

期望结果：

- 回答平台定位
- 不触发任何工具

### 场景 5.1.4：使用引导

输入示例：

- `怎么开始`
- `怎么用`
- `如何使用这个平台`

期望结果：

- 给出从图片/PDF/XML/CSV 开始的引导
- 不误触发执行

- 正确列出支持类型

### 场景 5.2.2：自动数字化 vs 手动数字化

输入示例：

- `手动数字化和自动数字化有什么区别`

前置条件：

- 已勾选 1 张心电图图片

期望结果：

- 不直接开始自动数字化
- 展示小选择卡

### 场景 5.3.3：PDF 已选，问“可以做什么”

前置条件：

- 已勾选 1 个心电图 PDF

输入示例：

- `这个 PDF 可以做什么`

期望结果：

- 不直接执行
- 展示适合 PDF 的选项

## 5.9 结果解释与多轮追问

### 场景 5.9.1：结果后追问“这是什么意思”

前置条件：

- 已产生工具结果

输入示例：

- `这是什么意思`

- `说明什么`

- 进入结果解释

输入示例：

- `危险吗`

- 可重新触发工具

## 5.10 用户乱问 / 非平台问答 / 闲聊

### 场景 5.10.1：完全无关问题

输入示例：

- `今天天气怎么样`
- `你会写诗吗`
- `讲个笑话`

期望结果：

- 不调用医疗工具
- 回到普通对话或温和兜底

### 场景 5.10.2：模糊垃圾输入

输入示例：

- `asdfgh`
- `....`
- `???`
- `123123`

期望结果：

- 不崩溃
- 不误触发任何工具

### 场景 5.10.3：恶搞或矛盾输入

输入示例：

- `不要分析，直接分析`
- `我只是问问，帮我立刻执行`

期望结果：

- 优先稳定，不应进入不可预测状态
- 若触发确认，也应可理解

### 场景 5.10.4：中英混输

输入示例：

- `this ecg image can analyze?`
- `这个 ECG csv can do what`

期望结果：

- 至少不应直接误执行
- 能识别则识别，不能识别时应安全兜底

## 5.11 用户误解平台能力

### 场景 5.11.1：要求直接给诊断结论

输入示例：

- `你直接告诉我是不是心梗`
- `你给我下诊断`

期望结果：

- 强调平台不能替代医生诊断
- 若有可用路径，可建议风险评估或数字化后分析

### 场景 5.11.2：没有波形却要求深度解读

前置条件：

- 仅有图片

输入示例：

- `直接给我做心电组学分析`

期望结果：

- 提示先数字化
- 不直接深度分析图片

## 5.12 连续多轮会话

### 场景 5.12.1：先问能力，再执行

会话示例：

1. `这张图可以做什么`
2. 点击 `自动数字化提取波形`

期望结果：

- 第一步展示小选择卡
- 第二步直接进入工具

### 场景 5.12.2：先执行，再追问，再重跑

会话示例：

1. `开始准心风险评估`
2. `这个结果什么意思`
3. `重新评估一次`

期望结果：

- 第一步执行
- 第二步解释结果
- 第三步重新执行

### 场景 5.12.3：切换来源后再问

会话示例：

1. 勾选图片，问 `可以做什么`
2. 改勾选 CSV，问 `这个可以分析吗`

期望结果：

- 第二步的推荐应切换为 CSV/XML 路由
- 不应沿用上一轮图片上下文误判

## 6. UI 与交互测试

### 场景 6.1：小选择卡样式

检查点：

- 出现在聊天区内部
- 不是全屏遮罩层
- 内容紧凑
- 保留加粗信息
- 功能项可点击

### 场景 6.2：确认态样式

检查点：

- 对需要确认的任务展示确认气泡
- 有明确“继续 / 只是问问”类操作

### 场景 6.3：缺失信息引导

检查点：

- 缺文件时有上传/选择引导
- 不只是报错

## 7. 稳定性与容错测试

### 场景 7.1：快速连续发送

操作：

- 连续发送 5 到 10 条不同问句

期望结果：

- 不崩溃
- 不出现明显串话

### 场景 7.2：中途取消

操作：

- 发送问题后立即取消

期望结果：

- 状态可恢复
- 后续还能继续对话

### 场景 7.3：长文本输入

输入示例：

- 带有长描述、长病史、长问题的输入

期望结果：

- 不崩溃
- 路由仍然稳定

### 场景 7.4：附件很多

前置条件：

- 上传多个来源文件

期望结果：

- 对话仍能工作
- 来源摘要不会导致明显混乱

## 8. 验收重点清单

以下项目建议作为每次发布前必过项：

1. 身份问答正确
2. 能力问答正确
3. `能分析吗` 不直接执行
4. 图片询问态显示小选择卡
5. 图片明确执行态直达工具
6. CSV/XML 明确执行态直达对应工具
7. 错文件类型不会误执行
8. 结果追问不会重跑工具
9. 无来源时有引导
10. 乱问不会误触发工具
11. 边界说明不越界，不替代医生
12. 小选择卡样式正常

---

## 9. 推荐执行顺序

如果时间有限，建议按以下顺序做：

### 第一轮：冒烟测试

1. 身份问答
2. 能力问答
3. 图片 + `能分析吗`
4. 图片 + `请自动数字化`
5. CSV + `请做心电组学分析`
6. 结果后 `这是什么意思`
7. 无来源 + `帮我分析`
8. 乱问 + `asdfgh`

### 第二轮：主流程测试

1. 图片 -> 小选择卡 -> 点击自动数字化
2. 图片 -> 准心风险评估 -> 结果追问
3. CSV -> 心电组学
4. CSV -> 信号补全
5. 图片误做心电组学的拦截

### 第三轮：异常与边界测试

1. 多文件混选
2. 中英混输
3. 长文本
4. 快速连续发送
5. 矛盾输入
